# Public API 兼容性检查（.NET）

## 1) 识别公共边界
- 列出本次涉及的 `public/protected` 类型与成员（按文件/命名空间）。
- 明确谁是调用方：外部应用/其他服务/SDK 用户/插件宿主等。

## 2) Breaking change 清单（逐项判定）
- 签名：方法/构造函数/泛型约束/可空性/可见性/virtual 语义。
- 行为：默认值、返回值含义、边界条件、线程安全保证。
- 错误语义：异常类型、错误码/结果对象语义、日志字段变化。
- 序列化：JSON/二进制/配置格式字段变化（兼容与迁移）。

## 3) 策略：弃用优于删除
- 优先新增替代 API + 标记旧 API `[Obsolete]`（写清替代项与移除计划）。
- 若必须破坏：给出迁移指南与最低版本升级建议（按项目需要最小化文档）。

## 4) 版本号建议
- 默认语义化版本：`MAJOR.MINOR.PATCH`。
- 不确定是否 breaking：按更严格处理（提高 MAJOR 或提供兼容层）。

## 5) 验证建议
- 至少 1 条：编译期（示例调用/测试）验证 + 运行期关键路径验证。
- 若有下游仓库：建议用最小样例或契约测试覆盖（按可用性）。
