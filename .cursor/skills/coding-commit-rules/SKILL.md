# 编码与提交守则（.NET）

在写代码或准备提交变更前，按本清单自检；适用于日常编码、提交 PR、或批量修改。

## 1) 目标与影响面（先写清楚）
- 用简短文字描述：改动目标、涉及模块/项目、对现有行为的预期影响（哪些不应变化）。

## 2) 代码卫生
- 变更后不遗留：死代码、注释掉的旧实现、无意义的临时日志、重复实现。
- 若替换了一条实现路径：同步删除旧路径或明确保留理由（例如灰度、兼容层、按配置切换）。

## 3) 异常与日志
- 禁止吞异常/静默失败；若捕获必须保留原异常，并记录关键上下文。
- 日志不得包含：密钥、令牌、密码、个人敏感信息（按项目合规要求）。

## 4) 兼容性与版本
- 若涉及 public API：识别 breaking change；优先“新增 + 弃用”。
- 若涉及序列化格式/配置项：必须考虑向后兼容与迁移。

## 5) 构建与测试
- 至少满足其一：`dotnet build` 通过 / `dotnet test` 通过（若有）/ 可执行的手工验证步骤。
- 对高风险改动给出回滚策略或降级开关（按项目需要）。

## 提交前快速核对
- [ ] 目标与影响面已写清
- [ ] 无死代码/无关日志/重复实现
- [ ] 无吞异常式兜底
- [ ] 无敏感信息泄漏
- [ ] 兼容性与版本策略明确（如相关）
- [ ] build/test 或手工验证可执行
