# 功能新增标准作业（.NET）

## 1) 先确认无现成实现
- 在决定“需要新增”之前，先在工程内搜索（类名、关键字、项目层次、扩展点）。
- 若已有实现：优先复用/扩展；说明为何能复用以及改动影响面。

## 2) 需求契约先行
- 明确输入/输出、错误语义、边界条件、不变量。
- 明确线程上下文要求（是否可在后台线程调用、是否要求同步上下文）。
- 若涉及配置：配置项名、默认值、迁移策略、可观测日志字段。

## 3) 路径与命名
- 遵循工程现有目录与命名风格；避免重复命名与“语义不清”的缩写。

## 4) 兼容性与依赖
- 若涉及公共 API：优先新增而非替换；弃用优于删除。
- 避免让公共 API 暴露第三方类型（减少依赖冲突扩散）。
- 若属于插件/宿主加载：显式考虑加载顺序、依赖冲突与隔离策略（按宿主能力）。

## 5) 验证
- 至少满足其一：`dotnet build`、`dotnet test`（若有）、或明确的手工验证步骤。
- 给出最小回归检查点，避免“新功能修好，旧功能坏了”。

## 新增前快速核对
- [ ] 已搜索确认无现成实现可复用
- [ ] 契约清晰（输入/输出/错误/边界/线程）
- [ ] 兼容性策略明确（特别是 public API）
- [ ] 依赖不会扩散冲突到调用方/宿主
- [ ] 有可执行验证步骤
